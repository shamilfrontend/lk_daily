## Полное техническое задание (финальная версия)

### 1. Введение

Разработать веб-приложение для управления очередностью демонстрации экранов на ежедневных созвонах в 11:00.  
В компании есть **несколько команд**, в каждой команде – свой список участников и своя циклическая очередь.  
Приложение должно автоматически определять, кто сегодня показывает экран, с учётом:

- отпусков сотрудников;
- нерабочих дней согласно Трудовому кодексу РФ (федеральные праздники, переносы выходных, региональные праздники);
- глобальных нерабочих дней, задаваемых администратором.

**Права доступа:**  
- **Один администратор** (учётная запись login/password) – полный CRUD всех сущностей (команды, участники, отпуска, нерабочие дни, ручная настройка очереди, отметка о выступлении/пропуске).  
- **Все остальные пользователи (неавторизованные)** – доступ **только на чтение** всех данных по **всем командам** (просмотр очереди, участников, отпусков, истории и т.д.). Редактирование невозможно.

### 3. Функциональные требования

#### 3.1. Аутентификация и права доступа

- **Страница входа** (`/login`) – поля: логин, пароль.  
- **Единственный администратор** – учётная запись создаётся автоматически при первом запуске приложения (из переменных окружения `ADMIN_LOGIN`, `ADMIN_PASSWORD`).  
- При успешном входе администратора выдаётся **JWT-токен** (время жизни 24 часа), который хранится в `localStorage`.  
- **Все GET-запросы** к API (чтение данных) – публичные, не требуют токена.  
- **Все POST/PUT/DELETE-запросы** – требуют валидный JWT-токен администратора (проверка через middleware). При отсутствии/невалидности – возвращается 401.  
- **Frontend**: если токен отсутствует или истёк – все UI-элементы редактирования (кнопки «Добавить», «Редактировать», «Удалить», drag-and-drop, формы) **скрыты или заблокированы**. Доступны только просмотр и навигация по командам.

#### 3.2. Управление командами (только админ)

- CRUD команд:  
  - Поля: `_id`, `name` (уникальное название), `description` (опционально), `region` (опционально – код региона для учёта региональных праздников, например `RU-MOW`).  
  - При удалении команды – все её участники **деактивируются** (isActive = false), запись очереди (`QueueOrder`) удаляется, история (`PresentationLog`) остаётся.  
- **Простые пользователи** видят список команд (выпадающий список или сайдбар), но не могут редактировать.

#### 3.3. Управление участниками (только админ)

- Поля: `_id`, `fullName`, `email` (опционально), `teamId` (ObjectId, ссылка на команду), `isActive` (boolean).  
- CRUD операции (создание, редактирование, мягкое удаление).  
- При добавлении нового активного участника в команду – он **автоматически добавляется в конец очереди** этой команды (`QueueOrder.userIds.push(userId)`).  
- При мягком удалении (`isActive = false`) – участник **удаляется из очереди** своей команды (фильтрацией массива `userIds`).  
- **Простые пользователи** видят список участников (можно фильтровать по команде), но не могут редактировать.

#### 3.4. Управление отпусками (только админ)

- Для каждого участника можно создать несколько периодов отпуска: `startDate`, `endDate` (включительно).  
- CRUD для отпусков.  
- **Простые пользователи** видят отпуска (просмотр), но не могут редактировать.

#### 3.5. Нерабочие дни – с учётом ТК РФ

Система должна определять нерабочие дни по следующим правилам:

##### 3.5.1. Федеральные праздничные дни (ст. 112 ТК РФ)
- 1–6, 8 января (Новогодние каникулы)
- 7 января (Рождество Христово)
- 23 февраля
- 8 марта
- 1 мая
- 9 мая
- 12 июня
- 4 ноября

##### 3.5.2. Переносы выходных дней (по постановлениям Правительства РФ)
- Если праздник выпадает на субботу или воскресенье – выходной переносится на следующий рабочий день.
- Для новогодних каникул: два дня, совпавших с выходными, переносятся на другие дни (обычно майские праздники или декабрь).  
- **Реализация**: хранить таблицу переносов `HolidayTransfer` с полями `fromDate`, `toDate`.  
- При определении нерабочего дня проверять: если дата является перенесённым выходным (`toDate`), то она нерабочая; если дата является исходным праздником, который перенесён (`fromDate`), то она **рабочая** (если не выпала на обычный выходной).

##### 3.5.3. Региональные праздники
- В модели `Team` добавлено поле `region` (например, `RU-TA` для Татарстана, `RU-BA` для Башкортостана).  
- В системе может быть таблица `RegionalHoliday`: `date`, `name`, `region`, `isWorkingDay: false`.  
- День считается нерабочим для команды, если существует региональный праздник с `region` = team.region.

##### 3.5.4. Пользовательские нерабочие дни (CRUD от администратора)
- Администратор может добавить любую дату как нерабочую (например, корпоративный выходной).  
- Эти дни хранятся в коллекции `NonWorkingDay` с типом `custom`.

##### 3.5.5. Алгоритм определения «рабочий ли день?» для заданной даты и команды
```typescript
function isWorkingDay(date: Date, teamRegion?: string): boolean {
  // 1. Если дата – пользовательский нерабочий день → false
  // 2. Если дата – перенесённый выходной (toDate в HolidayTransfer) → false
  // 3. Если дата – федеральный праздник, который НЕ перенесён (нет fromDate) → false
  // 4. Если дата – региональный праздник для региона команды → false
  // 5. Если дата – обычный выходной (суббота/воскресенье) и при этом она не является рабочим днём из-за переноса (fromDate) → false
  // 6. Иначе true
}
```

##### 3.5.6. API для нерабочих дней
- `GET /api/non-working-days?year=2026&teamId=...` – возвращает список всех нерабочих дней за год с указанием типа (`federal`, `transfer`, `regional`, `custom`).  
- `POST /api/non-working-days/batch` – массовый импорт производственного календаря (CSV или JSON) – только админ.  
- Стандартные CRUD для пользовательских нерабочих дней (добавление/удаление).  
- **Простые пользователи** могут только просматривать календарь нерабочих дней.

#### 3.6. Управление очередью демонстраций

##### 3.6.1. Порядок очереди
- Для каждой команды хранится **упорядоченный массив** `userIds` (в `QueueOrder`).  
- По умолчанию (при создании команды или при добавлении первого участника) порядок устанавливается по алфавиту (fullName) или по порядку добавления – на выбор администратора (в интерфейсе должна быть кнопка «Сортировать по алфавиту»).  
- Администратор может вручную изменять порядок через **drag-and-drop** (только для своей команды).

##### 3.6.2. Определение текущего докладчика на сегодня
- Получаем текущую дату (локальную, по серверу).  
- Если день нерабочий (по алгоритму выше) – докладчика нет, выводится сообщение «Сегодня нерабочий день, созвона нет».  
- Если день рабочий – берём порядок очереди команды (`userIds`).  
- Отфильтровываем участников, у которых есть отпуск, перекрывающий текущую дату (Vacation.startDate ≤ date ≤ Vacation.endDate).  
- Первый элемент после фильтрации – текущий докладчик. Если никого не осталось – возвращаем `null` (нет доступных).

##### 3.6.3. Отметка о выступлении (кнопка «Выступил»)
- Доступна только администратору (и только если сегодня рабочий день и есть докладчик).  
- Определяем текущего докладчика (по тому же алгоритму).  
- Перемещаем его ID из начала очереди в конец (`userIds.push(userIds.shift())`).  
- Сохраняем новый порядок в `QueueOrder`.  
- Записываем запись в `PresentationLog`: `{ teamId, date, userId, status: 'presented' }`.

##### 3.6.4. Пропуск выступления (кнопка «Пропустить»)
- Доступна только администратору.  
- Аналогично «Выступил», но в лог пишется `status: 'skipped'`.

##### 3.6.5. Защита от двойного нажатия
- Для одной даты и одной команды можно выполнить «Выступил» или «Пропустить» **только один раз** (проверка по наличию записи в `PresentationLog` за сегодня). При повторной попытке – возвращать ошибку 409.

##### 3.6.6. Прогноз очереди на будущие дни
- Эндпоинт `GET /api/queue/upcoming?teamId=...&days=7` – симуляция, которая возвращает для каждого из ближайших N рабочих дней (пропуская нерабочие дни) предполагаемого докладчика **без фактического изменения очереди**.  
- Симуляция: копия текущего порядка `userIds`, для каждого дня от today до today+maxDays:  
  - если день нерабочий – пропустить;  
  - если рабочий – найти первого доступного (не в отпуске) из текущей копии, удалить его из копии и добавить в конец, запомнить результат.

### 4. Модели данных MongoDB (Mongoose)

#### 4.1. Admin
```typescript
{
  _id: ObjectId,
  login: string,           // уникальный, например "admin"
  passwordHash: string     // bcrypt
}
```

#### 4.2. Team
```typescript
{
  _id: ObjectId,
  name: string,            // уникальное
  description?: string,
  region?: string,         // код региона, например "RU-MOW", "RU-TA"
  createdAt: Date,
  updatedAt: Date
}
```

#### 4.3. User
```typescript
{
  _id: ObjectId,
  fullName: string,
  email?: string,
  teamId: ObjectId,
  isActive: boolean,
  createdAt: Date,
  updatedAt: Date
}
```

#### 4.4. Vacation
```typescript
{
  _id: ObjectId,
  userId: ObjectId,
  startDate: Date,        // хранить дату без времени
  endDate: Date,
  createdAt: Date,
  updatedAt: Date
}
```

#### 4.5. NonWorkingDay (пользовательские + можно хранить все типы для удобства)
```typescript
{
  _id: ObjectId,
  date: Date,
  type: 'federal' | 'transfer' | 'regional' | 'custom',
  region?: string,         // для regional
  description?: string,
  createdAt: Date
}
```

#### 4.6. HolidayTransfer (переносы выходных)
```typescript
{
  _id: ObjectId,
  fromDate: Date,          // исходный праздник/выходной
  toDate: Date,            // день, который стал выходным
  year: number,
  description?: string
}
```

#### 4.7. QueueOrder
```typescript
{
  _id: ObjectId,
  teamId: ObjectId,        // уникальный индекс
  userIds: ObjectId[],     // порядок очереди (активные участники)
  updatedAt: Date
}
```

#### 4.8. PresentationLog
```typescript
{
  _id: ObjectId,
  teamId: ObjectId,
  date: Date,
  userId: ObjectId | null,
  status: 'presented' | 'skipped' | 'no_available',
  note?: string,
  createdAt: Date
}
```

### 5. API Endpoints (REST)

#### 5.1. Аутентификация
- `POST /api/auth/login` – тело `{ login, password }`, возвращает `{ token }`.  
- `GET /api/auth/verify` – проверка токена (опционально).

#### 5.2. Команды
- `GET /api/teams` – публичный  
- `POST /api/teams` – админ  
- `PUT /api/teams/:id` – админ  
- `DELETE /api/teams/:id` – админ

#### 5.3. Участники
- `GET /api/users?teamId=...&includeInactive=false` – публичный  
- `POST /api/users` – админ  
- `PUT /api/users/:id` – админ  
- `DELETE /api/users/:id` – админ (мягкое удаление)

#### 5.4. Отпуска
- `GET /api/vacations?userId=...&teamId=...&fromDate=...&toDate=...` – публичный  
- `POST /api/vacations` – админ  
- `PUT /api/vacations/:id` – админ  
- `DELETE /api/vacations/:id` – админ

#### 5.5. Нерабочие дни
- `GET /api/non-working-days?year=2026&teamId=...` – публичный (возвращает все нерабочие даты с типами)  
- `POST /api/non-working-days` – админ (добавить пользовательский)  
- `DELETE /api/non-working-days/:id` – админ  
- `POST /api/non-working-days/batch` – админ (массовый импорт)

#### 5.6. Переносы (для внутреннего использования)
- `GET /api/holiday-transfers?year=2026` – публичный  
- `POST /api/holiday-transfers` – админ (импорт переносов)  
- `DELETE /api/holiday-transfers/:id` – админ

#### 5.7. Очередь
- `GET /api/queue/current?teamId=...` – публичный  
- `GET /api/queue/order?teamId=...` – публичный (текущий порядок userIds)  
- `GET /api/queue/upcoming?teamId=...&days=7` – публичный  
- `POST /api/queue/present?teamId=...` – админ  
- `POST /api/queue/skip?teamId=...` – админ  
- `PUT /api/queue/order?teamId=...` – админ (тело: `{ userIds }`)

#### 5.8. История
- `GET /api/history?teamId=...&from=...&to=...&status=...` – публичный  
- `GET /api/history/export/csv?teamId=...&from=...&to=...` – публичный (скачать CSV)

### 6. Детали реализации

#### 6.1. Backend

- **Структура папок**:
  ```
  src/
    models/        (Mongoose модели)
    controllers/   (обработчики запросов)
    routes/        (Express роутеры)
    middlewares/   (authMiddleware, errorHandler)
    services/      (бизнес-логика: очередь, календарь, праздники)
    utils/         (dateHelpers, isWorkingDay)
    config/        (конфиг, env)
    seed/          (инициализация администратора)
  ```
- **Инициализация администратора**: при первом запуске, если коллекция Admin пуста, создать запись с login и хешем пароля из `ADMIN_LOGIN` и `ADMIN_PASSWORD` (env).  
- **JWT**: секрет из `JWT_SECRET`.  
- **Валидация**: Joi или class-validator для всех входных данных.  
- **Логирование**: morgan (запросы), winston (ошибки).

### 7. Алгоритм расчёта очереди (пример на псевдокоде)

```typescript
async function getCurrentPresenter(teamId: ObjectId, date: Date): Promise<User | null> {
  const team = await Team.findById(teamId);
  if (!team) throw new Error('Team not found');
  
  if (!isWorkingDay(date, team.region)) return null;
  
  const queueOrder = await QueueOrder.findOne({ teamId });
  if (!queueOrder || queueOrder.userIds.length === 0) return null;
  
  // Получаем всех активных пользователей команды
  const users = await User.find({ _id: { $in: queueOrder.userIds }, isActive: true });
  const vacations = await Vacation.find({ userId: { $in: users.map(u => u._id) }, 
                                          startDate: { $lte: date }, endDate: { $gte: date } });
  const userIdsOnVacation = vacations.map(v => v.userId.toString());
  
  // Ищем первого не в отпуске
  for (const userId of queueOrder.userIds) {
    if (!userIdsOnVacation.includes(userId.toString())) {
      return users.find(u => u._id.equals(userId));
    }
  }
  return null;
}
```

Функция `isWorkingDay` реализуется с использованием коллекций `NonWorkingDay`, `HolidayTransfer` и знания федеральных праздников (можно хранить их в коде или тоже в базе с типом `federal`).

### 8. Критерии приёмки

1. **Возможность создать несколько команд**, в каждой команде – свой список участников и своя очередь.  
2. **Администратор** может войти по логину/паролю (заданному через env) и получает доступ ко всем CRUD-операциям.  
3. **Неавторизованный пользователь** видит все данные (очередь, участников, отпуска, историю) по любой команде, но **не может ничего редактировать** (кнопки отсутствуют или заблокированы).  
4. **Очередь** правильно определяет текущего докладчика с учётом отпусков и нерабочих дней (федеральные праздники, переносы, региональные, пользовательские).  
5. **Кнопки «Выступил» / «Пропустить»** перемещают участника в конец очереди и записывают в историю. Одно нажатие в день на команду.  
6. **Прогноз на 7 дней** показывает корректных докладчиков, не изменяя реальную очередь.  
7. **Drag-and-drop** позволяет администратору менять порядок очереди для любой команды.  
8. **Производственный календарь** можно загрузить через CSV/JSON.  
9. **Экспорт истории** в CSV работает.  
10. Приложение запускается командами `yarn dev` (фронт) и `yarn dev` (бэк). Все зависимости указаны в `package.json`.  
11. Код покрыт типами TypeScript, ошибки обрабатываются, нет `any` (допустимо минимально).
